## AI-DLC とは

AWS が提唱している開発アプローチです。  
[AI 駆動開発ライフサイクル:ソフトウェアエンジニアリングの再構築](https://aws.amazon.com/jp/blogs/news/ai-driven-development-life-cycle/)  

チームコラボレーションで威力を発揮するものだとは感じますが、私は1人で利用しています。
1人でも有効的に使えますし、精度の高い(指示通りの)アプリケーションが出力されてきます。
重いプロセスでもって進行するのでちょっとした修正には向きませんが、モデルの進化もあり業務用途のアプリケーションを作るのには最適だと思います。

### 概要

(Claude に書いてもらいました)  

**AI-DLC (AI-Driven Development Life Cycle)** は、ソフトウェア開発のライフサイクルを AI アシスタントと人間チームで段階的に回すための構造化ワークフローです。「経験豊富なソフトウェアアーキテクトが隣にいる」感覚で、要件分析・設計・実装・テストを一貫したプロセスとして進めます。Kiro / Amazon Q Developer / Cursor / Cline / Claude Code / GitHub Copilot / OpenAI Codex といった主要な AI コーディングアシスタントに対応しています (本記事執筆時点の最新リリースは v0.1.8)。

特徴:

- ⚡ **適応的 (Adaptive Intelligence)** — リクエストに価値を加えるステージのみを実行
- 🧭 **コンテキスト認識 (Context-Aware)** — 既存コードベースと変更の複雑度を分析して工程を決定
- 🎯 **リスクベース (Risk-Based)** — 複雑な変更は包括的に、単純な変更は効率的に処理
- ❓ **質問駆動 (Question-Driven)** — 確認事項はチャットではなく、ファイル上の構造化された選択式質問として提示
- 🔍 **ユーザー主導 (Always in Control)** — 実行計画をレビューし、各フェーズを人間が承認してから着手
- 🧩 **拡張可能 (Extensible)** — セキュリティ・コンプライアンスなど組織独自のルールを extensions として追加可能 (セキュリティ / レジリエンシーのベースラインと、プロパティベーステストの extension が同梱)
- 📝 **完全な監査証跡** — 全ユーザー入力と AI の応答をタイムスタンプ付きで `aidlc-docs/audit.md` に、各ステージの実行 / スキップ状況を `aidlc-docs/aidlc-state.md` に記録
- 🚦 **承認ゲート** — Workspace Detection を除く各ステージの完了時に人間の明示承認を必須化 (Construction は "Request Changes" / "Continue to Next Stage" の 2 択)

### 開発フロー

AI-DLC は以下の 3 フェーズで構成されます。**ALWAYS** は必須、**CONDITIONAL** は条件付き、**PLACEHOLDER** は将来拡張用です。

```text
                       User Request
                            │
                            ▼
  ┌────────────────────────────────────────────────┐
  │  🔵 INCEPTION PHASE  (Planning & Architecture) │
  ├────────────────────────────────────────────────┤
  │  Workspace Detection        [ALWAYS]           │
  │  Reverse Engineering        [CONDITIONAL]      │  ← Brownfield のみ
  │  Requirements Analysis      [ALWAYS]           │  ← 深度は Minimal/Standard/Comprehensive で適応
  │  User Stories               [CONDITIONAL]      │
  │  Workflow Planning          [ALWAYS]           │
  │  Application Design         [CONDITIONAL]      │
  │  Units Generation           [CONDITIONAL]      │
  └────────────────────────────────────────────────┘
                            │
                            ▼
  ┌────────────────────────────────────────────────┐
  │  🟢 CONSTRUCTION PHASE  (Design → Build → Test)│
  ├────────────────────────────────────────────────┤
  │  (Per-Unit Loop)                               │  ← 1 ユニットを設計〜コード生成まで完了してから次のユニットへ
  │    Functional Design      [CONDITIONAL]        │
  │    NFR Requirements       [CONDITIONAL]        │
  │    NFR Design             [CONDITIONAL]        │
  │    Infrastructure Design  [CONDITIONAL]        │
  │    Code Generation        [ALWAYS]             │  ← Planning → Generation の 2 部構成
  │  Build and Test           [ALWAYS]             │  ← 全ユニット完了後に実施
  └────────────────────────────────────────────────┘
                            │
                            ▼
  ┌────────────────────────────────────────────────┐
  │  🟡 OPERATIONS PHASE   [PLACEHOLDER]           │
  └────────────────────────────────────────────────┘
                            │
                            ▼
                         Complete
```

主要ルール:

- **Code Generation は 2 部構成** — Part 1 (Planning) でチェックボックス付きの生成プランをユーザーが承認し、Part 2 (Generation) で承認済みプランに沿ってコードとテストを生成。テストの実行は後続の Build and Test ステージで行う
- **監査ログ** — ユーザー入力は**生テキストのまま**、ISO 8601 タイムスタンプと共に `aidlc-docs/audit.md` へ追記 (要約・言い換え・全体上書きは禁止)
- **2 レベルのチェックボックス追跡** — プランファイル内の即時マーク + `aidlc-docs/aidlc-state.md` での全体進捗
- **コードとドキュメントの分離** — アプリケーションコードはワークスペースルート、AI-DLC の成果物ドキュメントは `aidlc-docs/` のみに配置
- **Extensions の opt-in と強制力** — extensions の有効 / 無効は Requirements Analysis 中に opt-in プロンプトで選択。有効化した extension のルールは任意のガイダンスではなく強制制約となり、違反はステージ完了をブロックする (blocking finding)
- **NO EMERGENT BEHAVIOR** — 承認メニューに独自選択肢を足さない (Construction の完了メッセージは標準の 2 択固定)

詳細なルールは、各 AI アシスタント用に配置されるルールファイル (Claude Code の場合は `CLAUDE.md` と `.aidlc-rule-details/`) を参照してください。

## インストール

[AI-DLC](https://github.com/awslabs/aidlc-workflows) のリポジトリからルールファイルを自分の作業ディレクトリにコピーします。  
ルールファイルの集合体になっているので、決められた場所にファイル群をコピーするだけです。  

執筆日時点での Claude Code へのインストール手順は以下です。  

```bash
cp ~/Downloads/aidlc-rules/aws-aidlc-rules/core-workflow.md ./CLAUDE.md
mkdir -p .aidlc-rule-details
cp -R ~/Downloads/aidlc-rules/aws-aidlc-rule-details/* .aidlc-rule-details/
```

### Code Generation

コード生成が始まりました。今回はシンプルなアプリケーションかつローカル実行前提なのでさらりと進みましたが、デザインの複雑さによっては何回もデザインを繰り返すこともあるでしょうし、非機能やインフラのデザインが発生することもあると思います。

コード生成の成果物が `aidlc-docs/construction/demo-site/code/generation-summary.md` へ出力されていました。  

```markdown:aidlc-docs/construction/demo-site/code/generation-summary.md
# Code Generation Summary — Unit: demo-site

## 1. 生成ファイル一覧 (パス順)

## 2. FR ↔ ファイル トレーサビリティマトリクス

## 3. プラン違反 / 設計から逸脱した点

## 4. 確認すべき事項 (Build & Test ステージで)

## 5. 既知のデバッグ手段

## 6. 次ステージ (Build and Test) への引継ぎ

## 7. Post-Delivery Enhancement Session 1 — List Pagination
```

### Build and Test

コード生成が終わるとテストが実施されます。  
私の環境では以下のファイルが生成されました。

- build-and-test-summary.md: Coding Agent が実行した各種テストの結果サマリー
- build-instructions.md: 人間がテストを実施するために必要なビルドやテスト手順
- integration-test-instructions.md: 人間が結合テストを実施するためのテスト手順
- unit-test-instructions.md: 人間が単体テストを実施するためのテスト手順


このテストを人間が実行しながらバグや不具合を見つけて、Coding Agent に修正を依頼という流れを繰り返します。コードが修正されればテストも修正されます。
みなさんの想像通りですが、最初は60%ほどの精度です。サイクルを素早く回して完成まで導きます。  

コードの簡単な修正であれば、プロンプト、または、バイブコーディングで大丈夫です。ただ、仕様や設計に関わる修正は AI-DLC サイクルに則って修正します。こちらはプロセスが重いので敬遠されがちですが、仕様や設計の変更はドキュメントに残しておかないと後々破綻するので気持ちを切り替えて AI-DLC サイクルに戻りましょう。

## まとめ

様々なステークホルダーを集めて時間をかけて実施する AI-DLC を個人で使ってみました。  
私のような非開発者であっても要件や設計をまとめきれれば、それなりのアプリケーションを作ることが可能でした。  
重めのプロセスで少々の修正でも時間とトークンがかかりますが、これからの開発スタイルの進む道の1つではないでしょうか。  
物忘れが激しい Coding Agent を使うにあたり、ドキュメントを残していくことの大切さも改めて感じました。
